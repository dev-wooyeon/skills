# Problem Structuring

## 목적
복잡하게 얽힌 문제를 증상, 원인, 반복 요인, 해결 가능한 영역으로 나누어 구조화하는 스킬이다. 감각적으로 알고 있는 문제를 실행 가능한 형태로 바꿔준다.

## 이 스킬이 필요한 상황
문제가 큰데 어디부터 손대야 할지 모를 때
기술 문제와 조직 문제가 섞여 있을 때
문제를 설명은 할 수 있지만 문서화가 잘 안 될 때

## 핵심 질문
지금 보고 있는 것은 증상인가 원인인가
무엇이 문제를 반복시키는가
기술로 풀 수 있는 것과 정책으로 풀어야 할 것은 무엇인가
가장 먼저 해결해야 할 부분은 어디인가

## 입력 데이터
문제 상황 설명
관련 로그, 사례, 불만, 운영 이슈

## 분석 절차
1. 문제를 한 문장으로 정의한다.
2. 증상과 원인을 분리한다.
3. 반복 요인과 트리거를 정리한다.
4. 해결 가능한 영역을 기술, 프로세스, 운영으로 나눈다.
5. 우선순위를 매긴다.

## 산출물 형식
문제 정의
증상
근본 원인
반복 요인
영향 범위
해결 옵션
우선순위

## 평가 기준
문제를 말끔하게 쪼개서 설명할 수 있어야 한다.
문제 정의가 해결책보다 먼저 나와야 한다.
복잡한 상황을 공유 가능한 언어로 바꿔야 한다.

## 프롬프트 템플릿
```text
아래 문제를 구조화해서 정리해줘.

다음 항목으로 나눠줘.
1. 문제 정의
2. 현재 보이는 증상
3. 근본 원인
4. 문제가 반복되는 이유
5. 영향 받는 사용자 또는 시스템
6. 기술적으로 해결 가능한 부분
7. 프로세스로 해결해야 할 부분
8. 우선순위가 가장 높은 해결 과제
```

## 활용 예시
레거시 시스템의 유지보수 난이도 분석
운영 자동화 과제를 정의할 때

## 주의할 점
해결책부터 쓰기 시작하면 문제 정의가 흐려진다.
감정 표현보다 검증 가능한 사실 위주로 정리한다.
