# problem-structuring

## 역할

복잡하게 얽힌 문제를 증상, 근본 원인, 반복 요인, 영향 범위, 해결 가능한
영역으로 나누어 설명 가능한 형태로 바꾸는 스킬입니다.

## 언제 쓰는가

- 문제가 크지만 어디부터 손대야 할지 모를 때
- 기술 문제와 운영/프로세스 문제가 섞여 있을 때
- 장애나 운영 이슈를 문서화 가능한 형태로 정리해야 할 때

## 입력

- 문제 상황 설명
- 관련 로그, 사례, 운영 이슈

## 출력

- 문제 정의
- 증상
- 근본 원인
- 반복 요인
- 영향 범위
- 해결 옵션과 우선순위

## 핵심 규칙

- 문제를 한 문장으로 먼저 정의합니다.
- 해결책보다 증상과 원인을 먼저 분리합니다.
- 기술, 프로세스, 운영으로 해결 영역을 나눕니다.

## 관리 포인트

- 감정 표현보다 검증 가능한 사실 중심 원칙이 유지되는지 확인합니다.
- 산출물 형식이 실제 회고나 운영 정리 문서와 잘 맞는지 점검합니다.

## 실행 프롬프트

```text
$problem-structuring

문제 상황:
<현재 벌어지는 문제 설명>

관련 근거:
<로그, 사례, 운영 이슈, 사용자 불만 등>

위 문제를 아래 형식으로 구조화해줘.
1. 문제 정의
2. 현재 보이는 증상
3. 근본 원인
4. 문제가 반복되는 이유
5. 영향 받는 사용자 또는 시스템
6. 기술적으로 해결 가능한 부분
7. 프로세스로 해결해야 할 부분
8. 우선순위가 가장 높은 해결 과제
```
